Execute command.


The execute command is one of the hardest commands to master, but if you know how to use it it can be very powerfull. I'm here to tell you how the execute command works.


Subcommands.

The execute command is made up of subcommands, these are things you place after /execute so the command can only continue if the conditions are met. After your subcommands you put run and then the command you want to run as soon af the contidions are met.

Modify subcommands.
Modify subcommands are used to change the position of where the command is executed, this includes: align, anchored, at, facing, in, on, positioned and rotated. The as subcommand is also placed in this catagory, this subcommand changes the executer of the command. This is especialy usefull if you want to execute a command at an entity, to do this you type /execute as [target selector] at @s run [command].

Condition subcommands.

There are only two condition subcommands, if and unless. These on its own speak for itself, but if or unless what? Well the syntaxes for these subcommands are: biome, block, blocks, data, dimension, entity, loaded, predicate and score. Most of these are calculated from the place where the command is executed. Only score and predicate are not, score is true when a specified score from a scoreboard is a specified value. Predicate is true when the predicate itself is true, i am not going over predicates on this website.

Store subcommands.

The store subcommand is used to store specific data, this data can come from: blocks, entity's, tile entity's, bossbars and scores form a scoreboard. for blocks, entity's and tile entity's it can store the NBT data, so for this to work you have to specify wich data you want to get. For the bossbars and scores it is quite simple, once you specified witch score or bossbar you want to store it just stores the value of it.
